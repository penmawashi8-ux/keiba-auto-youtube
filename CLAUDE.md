# プロジェクトルール

## ライブラリ制約

### Pillow (PIL) は完全使用禁止
- **新規・既存コードを問わず** `from PIL import ...` / `import PIL` を一切使わないこと
- 過去のコードにPillowが残っていた場合も、触る機会があれば即座にffmpegに置き換えること
- 背景画像・フレーム画像の生成は **ffmpeg の lavfi/geq フィルター、drawtext、またはdrawbox** で行う
- サムネイルは **ffmpegのフレーム抽出**（`ffmpeg -ss 0.5 -i video.mp4 -vframes 1 thumbnail.jpg`）で生成する
- numpy も Pillow と一緒に使われていたため、同様に使用禁止

### ffmpeg で動画処理を行う際の注意
- `drawbox` はGitHub Actions環境で失敗するケースがあるため使用禁止
- テキスト背景パネルは `drawtext` の `box=1:boxcolor=...:boxborderw=N` オプションで実現する
- 日本語テキストは `text=` ではなく `textfile=` を使ってファイル経由で渡す（エスケープ問題を回避）
- ASS字幕の色は `&HAABBGGRR` 形式（ABGRの順）

### TTS (edge-tts) ボイス
- ニュースシリーズ: `ja-JP-KeitaNeural`（男性）
- 名馬列伝シリーズ: `ja-JP-NanamiNeural`（女性）
- `ja-JP-NaokiNeural` は存在しないため使用禁止

## YouTubeサムネイル設定について

このチャンネルは **YouTube Shorts（縦1080×1920）** を投稿している。

### 現在の状況（2026年6月17日更新）

**注意: 公式 `thumbnails().set()` API は信頼できない。HTTP 200 でも反映されない時期がある。**

2026年6月中旬に一度「コード変更なしでカスタムサムネイルが反映されるようになった」と
記録したが、同月17日に再びサムネイルが設定されなくなる事象が発生した。
Actions のログを確認したところ、`thumbnails().set()` 呼び出し自体は今回も
**エラーなく成功（HTTP 200）** している。つまりこのAPIは現在も
「200を返すが実際には無反応」という挙動に戻っている可能性が高い。

**ログだけでは「本当に反映されたか」を判定できない点に注意。**
サムネ抽出は動画の0.5秒目フレーム（`generate_video.py` でタイトルを
焼き込んだ先頭カード）をそのまま使っている。YouTube ShortsがAPI失敗時に
フォールバックで表示する「動画の最初のフレーム」も同じ画像になりうるため、
Studio の一覧でサムネにタイトルが乗って見えても、それがカスタムAPI経由で
設定されたものか、単にデフォルトの先頭フレームが表示されているだけなのかを
見た目だけで区別できない。本当に直っているか確認するには、YouTube Data API の
`videos.list`（`part=snippet`）でサムネイルURLを取得するか、Studioで
手動アップロード可否（電話番号認証エラーが出るか）を直接確認すること。

**今後この症状が起きたら：**
- まずは「コードや認証情報が変わっていないか」を確認する（変わっていないことが多い）
- Actions ログで `thumbnails().set()` の戻り値（200 or エラー）を確認する
- 200でも反映されないなら、これはYouTube側の仕様のフリップフロップであり、
  コード側で直す手段はない。Studio からの手動アップロードで回避する。

**重要: サムネイル画像は動画と同じ縦解像度（1080×1920）のまま送ること。**
ffmpegのフレーム抽出時に `-s 1280x720` などで横長にリサイズすると、
潰れた横長画像がそのままShortsのサムネイルとして表示されてしまう（2026年6月に発生・修正済み）。

正しい抽出コマンド:
```
ffmpeg -y -ss 0.5 -i video.mp4 -vframes 1 -q:v 2 thumbnail.jpg
```
（`-s` によるリサイズ指定は禁止。動画ネイティブの縦横比を維持する）

### 過去の調査記録（参考）

API が無効だった時代に試して全滅した方法:

| 方法 | 当時の結果 |
|---|---|
| 公式 `thumbnails().set()` API | HTTP 200 を返すが無反応（2026年6月、有効→再び無反応とフリップフロップ中） |
| YouTube Studio 内部API `youtubei/v1/video_manager/metadata_update` | HTTP 500 backendError（SAPISIDクッキーが必要と推測） |
| Playwright + ブラウザ自動操作 | GitHub Actions ではクッキーのセッションが維持できず断念 |

内部API・Playwright への再挑戦は基本不要（過去に全滅したため）。
ただし公式APIも現状では「200を返すだけで信用できない」状態なので、
過信せず都度実際の反映状況を確認すること。
